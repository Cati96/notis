1. Abstract

In this report we have highlighted the stages of the current project development, its description, the technologies used and the challenges encountered. Finally, we presented some conclusions and future improvements for our application.

2. Introduction and objectives

Notis is a web application that offers the possibility to search for different entities as notaries and translators based on different searching criterias such as: proximity, type of documents that can be released, etc.

The application offer real-time locations on map (provided by Google Maps) on which we can indentify the user location or actual positions of the offices and estimates the time needed to travel to the desired destination using different types of transport (personal vehicle, bicycle, public transport, walking).

For each type of entity we display informations regarding the documents that can be released, their prices and general contact informations such as: telephone number and email address. Also, we can check which are the additional documents we need to provide and we can download templates for them in order to complete them and to not waste any precious time.

4. RDF structure

RDF was generated using rdflib Python library. Data was provided in two excel documents and a Python script was used to filter the data, add missing information and to create a rdf document in the rdf/XML format.

The rdf contains two types of entities: Notary and Translator. Both entities are of type person. Their structure are very similar, only some minor differences are present. We can check the syntax for both entities below.

As we can see both entities have a similar structure, the only difference being the languages tag that is applying only for translators. This structure will suffer changes in the future, different types of document templates will be added and the prices for each service that is offered by an entity.

Different types of RDFs can be generated using the rdflib library such as:‘xml’, ‘n3’, ‘turtle’, ‘nt’, ‘pretty-xml’, ‘trix’, ‘trig’. The current format is pretty-xml.

6. Initial goals & Requirements

Our application starts from the premise that there are people who want to find out multiple information about notary officials from their country.

So, we established a couple of goals/requirements to follow in order to satisfy the end-users:

  • To can list all notaries and translators along with their base information like authorization number, address, timetable
  • For each official to display all services offered with their price and with all draft documents for each service
  • To search for notaries or translators in area of user's current location, or by selecting a custom location
  • Finally, from the end-user perspective to can search entities by their services offered

As a plus, to make our life easier, to can have an administrative console where we can manipulate all the notaries and translators.

7. Results

In the end, our application satisfies our initial requirements, so everybody can search for notary officials in order to find what kind of services they offer, which is their timetable, how they can be contacted and where they can be found.

More that this, we have a web administrative console, where just a few people can have access and in which we can add, remove or modify translators and notaries along with their information about services, documents, timetable and address.

8. Challenges encountered

One of our biggest challenge was to create the RDF model and to fill with the existent data about notary officials.

More than this, another difficulty was to manipulate the data inside the RDF, because we never worked with this kind of data modeling and interrogation.

Another challenge that we had was to work with the maps. Initially we started to work with GoogleMaps APIs and it seemed okay to work with this API until we wasn't able to use one of their feature because an amount of money was necessary to buy a key required for the API. So we had to change out maps API provider from Google Maps to something else. We tried multiple API providers after that until we found the OpenStreetMap API which was our lifeline.

We need some time to figure it out how the OSM API works, how we can work with the map container, overlayers, features, markers and other things, how to destroy in a good way, how to recreate them and so on.

9. Conclusion

To sum up, we think that we successfully accomplished our initial requirements so the end-users can benefit from our application if they want to find out more information about legalized notaries and translators.

For us, as a team, but not only, for us as individuals, we taught something really new, working with RDFs and SPARQL and another thing, how to implement a map container which has multiple features.

10. Future improvements

In the future we can add some new features like:
  • Adding for user possibility to add review and rating to one entity (notary/translator)
  • To offer more search criteria
  • To add on the maps possibility to see the best way to arrive to the entity's office with multiple transportation modes.
